## Phase 1: 逻辑规划 (Superpowers writing-plans)

**触发条件**: 任何复杂功能、架构变更、生产级实现

**强制输出**: `PLAN.md`

### 规划模板

```markdown

|---------|------|---------|
| `pkg/xxx/xxx.go` | 核心逻辑 | ~200 |
| `pkg/xxx/xxx_test.go` | 单元测试 | ~150 |

### 2.2 修改文件
| 文件路径 | 变更类型 | 影响范围 |
|---------|---------|---------|
| `pkg/yyy/yyy.go` | 接口扩展 | 新增方法 |

### 2.3 删除文件
| 文件路径 | 原因 |
|---------|------|
| 无 | - |

## 3. 逻辑设计

### 3.1 核心算法/流程
```
[流程图或伪代码]
```

### 3.2 状态机 (如有)
```
[状态转换图]
```

### 3.3 边界条件
| 场景 | 输入 | 预期输出 | 处理方式 |
|-----|------|---------|---------|
| 正常情况 | ... | ... | ... |
| 异常情况 | ... | ... | ... |
| 边界情况 | ... | ... | ... |

## 4. 接口设计

## Phase 2: 工程规范设计 (GStack /plan-eng-review)

**触发条件**: PLAN.md 获得用户确认

**评审维度**:

| 单例模式 | ⚠️ 谨慎 | 考虑依赖注入 |

### 模块职责
- [ ] 单一职责原则 (SRP)
- [ ] 开闭原则 (OCP)
- [ ] 依赖倒置原则 (DIP)

### 扩展性评估
- [ ] 是否支持水平扩展?
- [ ] 是否支持插件化?
- [ ] 配置是否外部化?

## Phase 5: CSO 安全评审 (Security Audit)

**评审重点**:

### 5.1 敏感信息检查
```markdown
## 安全检查清单

### 数据安全
- [ ] 无硬编码密码/密钥
- [ ] 无日志泄露敏感信息
- [ ] 配置文件权限正确 (0600)
- [ ] 环境变量读取安全

### 输入验证
- [ ] SQL 注入防护
- [ ] XSS 防护
- [ ] 命令注入防护
- [ ] 路径遍历防护

### 访问控制
- [ ] 身份认证
- [ ] 权限校验
- [ ] 最小权限原则
```

### 5.2 安全扫描规则
```yaml
security_rules:
  secrets:
    - pattern: "password\\s*=\\s*[\"'][^\"']+[\"']"
      severity: critical
    - pattern: "api_key\\s*=\\s*[\"'][^\"']+[\"']"
      severity: critical
  
  injection:
    - pattern: "fmt\\.Sprintf.*%s.*query"
      severity: high
    - pattern: "exec\\.Command.*user.*input"
      severity: high
```

### 5.3 合规检查
- [ ] 符合 OWASP Top 10
- [ ] 符合公司安全规范
- [ ] 符合行业合规要求 (如 GDPR)

## Phase 6: 编码实现 (Test-Driven Development)

**强制规则**:

### 6.1 TDD 流程
```
红 → 绿 → 重构
│     │      │
│     │      └── 优化代码结构
│     └───────── 编写最小实现
└─────────────── 编写失败测试
```

### 6.2 编码规范
- [ ] 每次提交 < 200 行
- [ ] 提交信息符合规范
- [ ] 测试先行
- [ ] 频繁运行测试

### 6.3 代码审查清单
```markdown
## 自审查清单

### 功能完整性
- [ ] 实现了 PLAN.md 中的所有功能点
- [ ] 处理了所有边界条件
- [ ] 错误处理完善

### 代码质量
- [ ] 函数长度 < 50 行
- [ ] 圈复杂度 < 10
- [ ] 无重复代码
- [ ] 命名清晰

### 测试覆盖
- [ ] 单元测试覆盖率 >= 80%
- [ ] 关键路径 100% 覆盖
- [ ] 集成测试通过
```

## 强制阻断规则

以下情况必须阻断流程，返回修改：

| 阶段 | 阻断条件 |
|-----|---------|
| Phase 1 | 未生成 PLAN.md / 未识别边界条件 |
| Phase 2 | 技术栈不合规 / 架构设计有缺陷 |
| Phase 3 | 存在循环依赖 / 接口设计不合理 |
| Phase 4 | 测试覆盖率目标未设定 / 边界条件未识别 |
| Phase 5 | 发现安全漏洞 / 敏感信息泄露风险 |
| Phase 6 | 测试未通过 / 代码规范检查失败 |
| Phase 7 | 覆盖率不达标 / 有 race condition |
