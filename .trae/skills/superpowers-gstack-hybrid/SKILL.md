## 核心概念定义

| 概念        | 定义                                     |
| --------- | -------------------------------------- |
| **变更文件数** | 新增、修改、删除的文件总数（不含自动生成文件）                |
| **新增代码行** | 不含空行和注释的净增代码行数                         |
| **架构影响**  | 涉及模块间接口、数据流、部署方式的变更                    |
| **风险等级**  | 低：仅影响非核心功能；中：影响核心功能但可回滚；高：涉及数据迁移或不可逆变更 |
| **接口变更**  | 对已有公共 API 的签名、参数、返回值的修改                |
| **依赖变更**  | 新增或替换 go.mod 中的外部依赖                    |

## 任务复杂度分级

根据变更范围、影响面和风险等级，将任务分为三级：

### L1 - 简单任务 (轻量级流程)

**判定标准** (满足任一):

- 修改文件 < 3 个
- 新增代码 < 100 行
- 纯 bug 修复，无架构变更
- 文档更新

**简化流程**:

Phase 1 (简化) → Phase 6 → Phase 7 (简化)

**跳过**: Phase 2-5 的正式评审，改为快速自检查

### L2 - 中等任务 (标准流程)

**判定标准** (满足任一):

- 修改文件 3-8 个
- 新增代码 100-500 行
- 新增接口或模块
- 影响现有功能

**标准流程**:

Phase 1 → Phase 2-3 → Phase 6 → Phase 7

**跳过**: Phase 4-5 (QA/CSO 评审合并到 Phase 7)

### L3 - 复杂任务 (完整流程)

**判定标准** (满足任一):

- 修改文件 > 8 个
- 新增代码 > 500 行
- 架构重构
- 涉及安全或性能关键路径

**完整流程**:

Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5 → Phase 6 → Phase 7

&#x20; │

&#x20; ├─ Phase 4 不通过

&#x20; │   └─ 边界条件遗漏 → 补充边界处理 → 重新提交 Phase 4

&#x20; │

&#x20; ├─ Phase 5 不通过

&#x20; │   └─ 敏感信息泄露 → 清理敏感数据 → 重新提交 Phase 5

### 异常处理原则

| ----- | ----------------------- |
| 小范围修复 | 仅修改不通过的部分，不扩大变更范围       |
| 记录原因  | 每次不通过必须记录原因和修改方案        |
| 限制重试  | 同一 Phase 连续 3 次不通过，升级处理 |
| 回溯检查  | 修改后需确认不影响已通过的 Phase     |

## 附录 A: 术语表

| 术语    | 全称                                                  | 说明                   |
| ----- | --------------------------------------------------- | -------------------- |
| SRP   | Single Responsibility Principle                     | 单一职责原则：一个类/模块只负责一项职责 |
| OCP   | Open-Closed Principle                               | 开闭原则：对扩展开放，对修改关闭     |
| DIP   | Dependency Inversion Principle                      | 依赖倒置原则：依赖抽象而非具体实现    |
| ISP   | Interface Segregation Principle                     | 接口隔离原则：客户端不应依赖不需要的接口 |
| LSP   | Liskov Substitution Principle                       | 里氏替换原则：子类型必须能替换其基类型  |
| TDD   | Test-Driven Development                             | 测试驱动开发：先写测试再写实现      |
| DRY   | Don't Repeat Yourself                               | 不要重复自己：避免代码重复        |
| SOLID | SRP+OCP+LSP+ISP+DIP                                 | 面向对象设计五大原则的统称        |
| OWASP | Open Web Application Security Project               | 开放式 Web 应用安全项目       |
| CSO   | Chief Security Officer                              | 首席安全官                |
| QA    | Quality Assurance                                   | 质量保证                 |
| CR    | Code Review                                         | 代码审查                 |
| KPI   | Key Performance Indicator                           | 关键绩效指标               |
| OOM   | Out of Memory                                       | 内存溢出                 |
| SSRF  | Server-Side Request Forgery                         | 服务端请求伪造              |
| HSTS  | HTTP Strict Transport Security                      | HTTP 严格传输安全          |
| GDPR  | General Data Protection Regulation                  | 通用数据保护条例             |
| HIPAA | Health Insurance Portability and Accountability Act | 健康保险流通与责任法案          |
| P99   | 99th Percentile                                     | 第 99 百分位延迟           |
| E2E   | End-to-End                                          | 端到端测试                |
